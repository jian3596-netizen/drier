# 控温与安全评估 / 改进意见

> **文档性质**：基于**源码 + 原理图**的设计评审（design review），**未经台架实测**。凡标注"⚠ 需实测"的结论，必须上硬件验证后才能定论。
> **日期**：2026-06-08
> **范围**：烘干器项目，STM32F030F4P6 控制板 + 迪文 T5L 屏。
> **核心问题**：① 当前电路/固件能否加热到 90℃；② 有无安全风险。
>
> **🔄 更新（2026-06）**：STM32 控温固件已按本评估**重写**——风险 1/3/4/5 已在固件加入安全联锁（**待台架验证**）；风险 2（独立硬件过温）仍需硬件改动。各项见下方 ✅ 标注。

---

## 〇、先记住一个前提：控温是"两颗芯片分工"

- **测温 + 算温度 + 给目标**：在**屏端 8051**（`2_屏幕固件_8051/HANDWARE/TASK/task.c` 的 `GET_Temp` 查表，含校准系数 `Set_Temp_xs`）。
- **真正控加热/风扇的 PWM**：在 **STM32**（重写后位于 `1_STM32固件_控制板/USER_APPLICATION/control.c`）。
- 两者靠 UART(115200) 通信。**这条链路任何一环断了，控温就失效**——这是后面多条风险的根源。

加热 PWM 极性是**反的**：`TIM16 比较值 99 = 关，0 = 全功率，值越小越热`。（改代码极易搞错，务必牢记。）

## 一、能力评估：能到 90℃ 吗？

### 1.1 测温到 90℃ —— ✅ 支持
- 屏端温度表覆盖 **0~130℃**，90℃ 在范围内。
- NTC 6.8k 分压，90℃ 时 NTC≈4.5kΩ，与 6.8k 同量级，这一段分辨率良好。
- 设定范围本就是 0~99℃，90℃ 属设计内。

### 1.2 加热到 90℃ —— ⚠ 设计意图支持，实际需实测
电气驱动能力充足（NCE6050 = 60V/50A MOS，余量很大）。但**能否真的升到 90℃**取决于三个"图上定不了"的量：
| 变量 | 说明 | 在哪确认 |
|---|---|---|
| 发热体功率 | 外接，插 U3/U8 接口，原理图无规格 | 查发热体规格书 |
| 风量 | 风扇把热量带走，风量越大越难升温 | 实测 |
| **VBUS 实际电压** | 板上有 `K7805` DC-DC，说明 VBUS **很可能 >5V**（如 12V）；加热功率≈VBUS²/R | 测 VBUS 电压 |

**结论**：90℃ 在设计范围内，但是否可达**必须上台架实测**，不能只看图下结论。

## 二、安全风险清单（按严重程度排序）

| 级别 | 风险 | 固件处置（2026-06 重写后）|
|---|---|---|
| 🔴 高 | 通信中断不失效安全，加热可能卡在全开 | ✅ 已加通信超时(>1s)关加热（待台架验证）|
| 🔴 高 | 无独立硬件过温保护，全靠固件 | ❌ 仍需**硬件**温控开关/温度保险丝（软件替代不了）|
| 🔴 高 | 无看门狗(IWDG/WWDG)，死机不复位 | ✅ 已启用 IWDG(~3.2s)（待台架验证）|
| 🟠 中 | NTC 探头脱落→读冷→可能满功率加热 | ✅ 已加本地电阻开路/短路检测（阈值待台架核实）|
| 🟡 低 | 开关式控制(bang-bang)温度过冲 | ✅ 已改双区 PI（增益待整定）|

> 下面各风险的详细分析保留作**原理依据**；"固件处置"即上表。风险 2 是唯一**软件解决不了**的，务必推动硬件。

### 🔴 风险1：通信中断不会"失效安全"
**现象**：加热 PWM **只在收到屏的串口包时更新**（`main.c:150` 的 `if(st_Uart1.USART_RX_SUCCESS==1)` 内，所有 `SetCompare(&htim16,...)` 都在这个 if 里）。
**后果**：FPC 排线松动 / 屏死机 → STM32 收不到包 → **PWM 保持上一次值**。若断线瞬间正处于"全功率加热(0)"，则**加热一直全开，且无温度反馈**。
**为什么严重**：这是加热失控最可能的真实触发路径（排线接触不良在手持设备里很常见）。

### 🔴 风险2：没有独立的硬件过温保护
**现象**：关加热的判断**全在固件**。电路上只有 **PPTC 自恢复保险（F1/F2）**。
**关键**：PPTC **只防过流，不防过温**。没有温度保险丝、机械温控开关(KSD)、或独立比较器做后备。
**后果**：固件一旦失效（风险1/3），**没有任何独立后备**阻止加热。

### 🔴 风险3：没有看门狗
**现象**：已确认 IWDG/WWDG 均未启用（`stm32f0xx_hal_conf.h` 里只有宏开关，无 `MX_IWDG_Init` 调用）。
**后果**：MCU 死机卡在加热状态 → **不会自动复位停机**。
**注**：上电默认 `TIM16=99`(关，`main.c:109`)是对的，所以**复位本身是安全的**——只要能复位。看门狗正是用来触发复位的。

### 🟠 风险4：NTC 探头脱落 → 读冷 → 可能满功率加热
**原理**：电路分压为 `3.3V — NTC — 节点(ADC) — 6.8k — GND`（NTC 在高侧）。
- 探头**脱落/开路** → 节点被 6.8k 拉到 GND → ADC≈0 → `ADC_DRIVE.c` 算出电阻极大 → `GET_Temp` 查表返回 **0℃（偏冷）** → 控温判定"没到温"→ **满功率加热**。
- 固件里 `温度>128℃` 的保护只能挡住**短路**（读高温），**挡不住开路**。
**⚠ 需实测**：上台架拔掉 NTC 探头，观察是否满功率加热来确认。（电路拓扑支持此担忧，但 OCR 自原理图，需实物核对分压方向。）

### 🟡 风险5：开关式控制温度过冲
**现象**：只有 1℃ 滞回 + 全开/全关（非 PID）。到温判据 `≥设定+1` 关、`≤设定` 开。
**后果**：叠加热惯性，**实测温度会冲过设定值**；90℃ 设定可能瞬时更高。影响精度，也吃掉外壳/元件的安全余量。

## 三、改进措施

### 3.1 固件加固 —— ✅ 已实现（位于 `USER_APPLICATION/control.c`，待台架验证）

> 下面 ①~⑤ 均已落地到 `control.c`。**所有可调参数集中在 `control.c` 顶部的 `g_cfg` 配置结构体**（RAM 全局，不是 `#define`——可在 J-Link 调试器 Watch 窗口**实时修改、不必重新编译**，相当于配置文件）：`plate_temp_max`(加热片过温, 默认 **128**)、`hard_temp_max`(单区过温, 120)、`res_min_ohm`/`res_max_ohm`(NTC 短路/开路)、`pi_kp`/`pi_ki_inc`(PI 增益)。默认值均为**占位值，需台架整定**。下面的"示意代码"保留作设计思路。

**① 通信超时关加热（对应风险1）** — 最优先
- 思路：记录上次成功收包的时间戳（用 SysTick / `HAL_GetTick()`）。主循环检查：超过阈值（如 **1 秒**）没收到包 → 强制 `TIM16=99`(关) + 风扇按需。
- 示意：
  ```c
  // 收到包时：last_rx_tick = HAL_GetTick();
  // 主循环每轮：
  if (HAL_GetTick() - last_rx_tick > 1000) {   // 1s 无通信
      __HAL_TIM_SetCompare(&htim16, TIM_CHANNEL_1, 99);   // 关加热
      // 可选：关风扇 / 点故障灯
  }
  ```

**② 启用 IWDG 看门狗（对应风险3）**
- CubeMX(`HGJ.ioc`) 里使能 IWDG，超时设为如 **1~2 秒**；主循环正常时喂狗(`HAL_IWDG_Refresh`)。
- 死机 → 不喂狗 → 自动复位 → 上电默认加热关(已确认) → 安全。
- ⚠ 注意：喂狗要放在"程序正常流转"的路径上，**不能**无脑每轮都喂（否则死循环也会喂狗，失去意义）。

**③ 开路/异常检测（对应风险4）**
- 在控温前增加合理性检查：温度读数 `≤0℃` 或 电阻超出合理区间 → 判为传感器故障 → 关加热 + 报警。
- 把现有 `>128℃`(短路保护) 扩展为**双向**：过高(短路) 和 过低/开路 都关加热。
- ⚠ 实测确认开路到底读什么值后，再定阈值。

**④ 控温改进（对应风险5，非安全必需，提精度）**
- 短期：加大滞回（如 ±2~3℃）减少频繁通断；或加最小开/关时间。
- 长期：把 bang-bang 升级为 **PID / PI** 控制（用 TIM16 的占空比按温差比例调功率），过冲会明显改善。

**⑤ 加热片过温保护（恢复原固件逻辑）** ✅
- 加热片传感器（温度3 / `temp3`）≥ `g_cfg.plate_temp_max`（默认 **128℃**）→ **切断两路加热**，风扇继续排热，保护加热片与外壳。
- 对应原固件的 `温3>128 → 关加热` 三路硬切；屏端在温度 >128 时也会弹窗报错。
- ⚠ 局限：加热片 NTC **开路**时温度3 读 ~0℃（偏冷），此保护看不见——只防"真过热"，防不了"探头掉了"。独立硬件后备仍见风险2。

### 3.2 硬件改进（需与画板的人商量）
- **强烈建议**在发热体回路串一个**机械温控开关(KSD301 之类)或温度保险丝**做**独立硬件后备**——与 MCU 无关，固件失效也能切断加热。这是加热类产品安规(如 CCC/CE)通常的硬性要求，**纯软件保护一般过不了认证**。
- 评估发热体功率与外壳耐温，确认 90℃ 工况下外壳可触及表面温度符合安规限值。

### 3.3 顺带修的代码隐患（已清理）
- `HC595_DRIVE.c`（板上无 595）、`Temperature.c`（STM32 端不算温度）、任务框架等遗留无效代码：本次重写已从工程移除。
- 微动开关 / 插入检测：**该功能产品已取消**，STM32 端相关 `KEY1/KEY2`（PF1/PF0）GPIO 定义与初始化已删除（`gpio.c` / `main.h`）。
  - ⚠️ 注：CubeMX 工程 `HGJ.ioc` 中 PF0/PF1 仍标为 KEY1/KEY2。**若日后用 CubeMX 重新生成代码，需在 GUI 里一并删掉这两个引脚**，否则会被重新生成回来。
  - 屏端 8051 仍保留对 SW1/SW2 字节的读取，但不参与控制、读到恒为 0，属历史遗留、不影响功能。
